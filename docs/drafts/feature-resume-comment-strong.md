좋습니다. 이 추가 기능은 매우 좋습니다.
기존의 **텍스트 주석 + 질문 카드** 기능을 한 단계 더 발전시켜서,
**실전 면접에서 실제로 많이 공격받은 이력서 포인트를 시각적으로 드러내는 “이력서 면접 히트맵” 기능**으로 설계하면 됩니다.

이 기능이 들어가면 사용자는 이력서를 그냥 문서로 보는 것이 아니라,

* 어디에서 질문이 가장 많이 나왔는지
* 어떤 문장이 반복적으로 검증되었는지
* 어떤 항목이 꼬리질문을 많이 유발했는지
* 어떤 부분은 거의 질문되지 않았는지

를 한눈에 볼 수 있게 됩니다.

# 3. 왜 이 기능이 중요한지

이 기능의 핵심 가치는 단순 시각화가 아닙니다.
사용자가 실제 면접 경험을 통해 **“이력서에서 어떤 문장이 리스크이자 기회인지”**를 바로 파악하게 해줍니다.

예를 들면 아래와 같습니다.

* 어떤 프로젝트 문장은 면접관이 거의 반드시 물어봅니다.
* 어떤 성과 문장은 숫자가 있어도 계속 검증 질문이 붙습니다.
* 어떤 기술 키워드는 표면 질문이 아니라 깊은 꼬리질문으로 이어집니다.
* 반대로 사용자는 중요하다고 생각했는데 실제로는 거의 질문되지 않는 부분도 있습니다.

즉, 이 기능은
**이력서의 “질문 유발도”를 시각화하는 기능**입니다.

# 4. 핵심 컨셉

이 기능은 기본적으로 **이력서 위에 실전 면접 데이터가 덧씌워진 레이어**입니다.

## 기본 구조

* 기준 문서: 특정 이력서 버전
* 연결 데이터: 실전 면접 기록들
* 분석 결과:

    * 질문이 나온 문장
    * 질문 횟수
    * 꼬리질문 수
    * 질문 유형
    * 압박 강도
    * 최근 질문 시점
* 시각화:

    * 옅은 음영: 가끔 질문 나온 부분
    * 짙은 음영: 자주 질문 나온 부분
    * 별도 뱃지: 꼬리질문이 많이 붙은 부분
* 클릭 시 질문 카드 목록

# 4A. 현재 구현과 다음 확장 단계

현재 백엔드 기준으로는 히트맵이 주로 **parsed resume anchor 단위**로 동작합니다.

예:

* 프로젝트 전체
* 경력 한 줄
* 기술 태그
* competency 블록
* summary 블록

이 방식은 첫 단계로는 유효하지만,
사용자께서 원하시는 UX는 여기서 한 단계 더 나아간 **문장 단위 / 구문 단위 오버레이**입니다.

즉, 앞으로는 아래 두 레이어를 함께 가져가는 것이 맞습니다.

## Layer 1. Anchor Heatmap

프로젝트 전체나 경력 전체처럼
“이 블록 전반에서 질문이 많이 나왔다”를 보여주는 레이어

## Layer 2. Text Range Overlay

프로젝트 설명 안의 특정 문장이나 구절에
실제 질문 카드와 코멘트가 직접 달리는 레이어

예:

* 프로젝트 카드 전체는 연한 배경으로 강조
* 그 안의 특정 성과 문장은 더 진하게 강조
* hover 시 해당 문장에 연결된 질문 카드 popover 표시

이 구조가 가장 자연스럽습니다.

# 5. 시각화 방식

## A. 기본 음영 강도

질문 빈도가 높은 부분일수록 더 짙게 표시합니다.

예시:

* 1회: 아주 연한 노랑
* 2~3회: 중간 노랑
* 4~6회: 진한 노랑/주황
* 7회 이상: 더 짙은 강조색

이 방식은 직관적입니다.

---

## B. 질문 성격에 따른 추가 시각 요소

빈도만으로는 부족할 수 있으므로, 질문의 성격도 같이 보여주는 것이 좋습니다.

### 예시

* **자주 나온 질문**: 색 농도 진하게
* **꼬리질문이 많음**: 점선 테두리 또는 작은 가지 아이콘
* **압박성 검증 질문 비중 높음**: 경고 아이콘 또는 붉은 계열 보조 뱃지
* **최근 면접에서 반복 등장**: 작은 “recent” 마커
* **답변 품질이 낮았던 포인트**: 약점 태그 표시

즉, 색 하나만으로 끝내기보다
**농도 + 아이콘 + 뱃지** 조합이 더 좋습니다.

---

## C. 문단 단위 / 문장 단위 / 키워드 단위 강조

이력서에서는 질문이 항상 한 줄 전체에서만 나오지 않습니다.
특정 키워드나 특정 성과 문장에 집중되기도 합니다.

그래서 3단계 레벨이 좋습니다.

### 1) 문단 단위 히트맵

프로젝트 전체 섹션이 많이 질문된 경우

### 2) 문장 단위 히트맵

특정 성과 문장이 많이 질문된 경우

### 3) 키워드 단위 핀포인트 강조

예:

* Redis
* Kafka
* 대규모 트래픽
* 성능 개선
* 장애 대응

이 정도까지 지원하면 활용성이 높습니다.

### 실무 추천

실제 구현은 아래 우선순위가 좋습니다.

1. 프로젝트/경력 block 단위 heatmap
2. 문장 단위 highlight overlay
3. phrase/keyword 단위 micro-highlight

즉, “전체 프로젝트에 대한 질문”과 “이 문장에만 붙는 질문”을 동시에 보여줘야 합니다.

예:

* “이 프로젝트 전반에 대한 질문 6개”
* “이 문장에 대한 검증 질문 3개”
* “이 수치 표현에 대한 꼬리질문 2개”

# 5A. 원하는 화면 동작 정의

사용자께서 보여주신 화면 기준으로는 아래 동작이 핵심입니다.

## 1) 프로젝트 전체 질문

프로젝트 제목, 요약, 역할, 기술, 내용 전체 중
특정 한 문장으로 좁히기 어려운 질문은 **project-level anchor**에 붙습니다.

예:

* “이 프로젝트에서 가장 어려웠던 점이 무엇이었나요?”
* “전체적으로 어떤 아키텍처 결정을 내렸나요?”

이 질문은 프로젝트 카드 전체에 연결됩니다.

## 2) 특정 문장 질문

예를 들어 내용 영역 안의

* “콘텐츠 라이프사이클 파이프라인 구조화 및 운영 안정화”

같은 문장에 대해 나온 질문은
그 문장 range에 직접 붙어야 합니다.

이 경우 UX는 아래가 적합합니다.

* 문장 하이라이트
* hover 시 질문 카드 preview popover
* 클릭 시 우측 패널 또는 상세 카드

## 3) 여러 문장/구절에 걸친 질문

질문이 한 문장이 아니라 두세 문장을 합쳐서 나온 경우도 있습니다.
이때는 contiguous range를 허용해야 합니다.

예:

* 앞 문장의 문제 상황
* 다음 문장의 해결 방식

을 함께 보고 물은 질문이라면, 두 문장을 하나의 overlay target으로 묶습니다.

# 5D. 왜 overlay target이 필요한가

현재처럼 anchor-level만 있으면 아래 문제가 있습니다.

* 프로젝트 전체가 질문되었다고만 보임
* 어떤 문장이 실제 trigger였는지 hover UX를 만들기 어려움
* “전체 질문”과 “문장 질문”을 구분하기 어려움

overlay target이 있으면 아래가 가능합니다.

* 프로젝트 전체 질문은 `targetType = block`
* 특정 문장 질문은 `targetType = sentence`
* 특정 기술 태그 질문은 `targetType = keyword`

즉, 같은 프로젝트 안에서도 질문의 초점을 더 정확히 표현할 수 있습니다.

# 6. 어떤 데이터를 기반으로 음영 강도를 정할지

단순 질문 횟수만 쓰기보다는 **가중치 기반 점수**가 더 좋습니다.

## 추천 점수 요소

### 1) 직접 질문 수

해당 문장과 연결된 질문 수

### 2) 꼬리질문 수

그 질문에서 이어진 follow-up 수

### 3) 압박 강도

검증형, 반박형, 깊이 파고드는 질문이면 가중치 부여

### 4) 면접 수 기준 반복성

한 번의 면접에서 5번 나온 것보다
서로 다른 여러 면접에서 반복 등장한 것이 더 중요할 수 있습니다.

### 5) 최근성

최근 면접일수록 조금 더 높은 가중치

### 6) 답변 난이도/취약도

사용자가 자주 약했던 지점이면 추가 강조 가능

---

## 예시 점수 공식

예를 들어 내부적으로는 이런 식으로 계산할 수 있습니다.

`heatScore = directQuestions * 1.0 + followUps * 1.5 + pressureQuestions * 2.0 + distinctInterviewCount * 1.2`

그리고 이 점수를 기준으로 색 농도를 나눕니다.

이렇게 하면 더 실전적인 시각화가 됩니다.

# 7. 사용 시나리오

## 시나리오 1: 이력서 복기

사용자가 특정 이력서 버전을 엽니다.
앱이 실전 면접 데이터와 매칭해 히트맵을 보여줍니다.

보이는 것:

* “이 프로젝트는 자주 질문됨”
* “이 문장은 꼬리질문 유발 빈도 높음”
* “이 수치 표현은 반복 검증됨”

사용자는 어디를 더 준비해야 하는지 즉시 알 수 있습니다.

---

## 시나리오 2: 질문 카드 확인

사용자가 짙은 음영 부분을 클릭합니다.

열리는 패널:

* 이 문장에서 실제로 나왔던 질문 목록
* 질문 빈도
* 최근 면접에서 나온 예시
* 연결된 질문 카드
* 예상 새 꼬리질문
* 다시 연습하기 버튼

이 구조가 가장 중요합니다.
음영은 단순 장식이 아니라 **질문 카드로 진입하는 입구**여야 합니다.

---

## 시나리오 3: 이력서 첨삭과 연결

사용자가 히트맵을 보고 특정 문장을 선택합니다.

예:

* “API 응답 속도를 80% 개선”

짙은 음영으로 표시됨
→ 실제 질문:

* 어떻게 측정했습니까?
* 어떤 병목이었습니까?
* 왜 그런 구조가 느렸습니까?
* Projection/Lazy/IN 절 기반 조회를 어떻게 조합했습니까?

사용자는 이 문장을 더 구체적으로 다듬거나,
질문 카드에 대한 답변을 별도로 준비할 수 있습니다.

---

# 8. 질문 카드와의 연결 방식

이 기능은 반드시 기존 질문 카드 시스템과 합쳐져야 합니다.

## 연결 구조

음영 처리된 텍스트 영역에는 다음이 붙을 수 있습니다.

* 실제 실전 질문 카드
* AI 확장 질문 카드
* 꼬리질문 카드
* 첨삭 코멘트
* 답변 개선 메모

즉, 하나의 강조 영역은 아래를 담는 “복합 노드”가 될 수 있습니다.

### 예시

선택 영역:
“MapStruct 매핑 검증을 통해 잘못된 객체 진입을 차단”

연결 정보:

* 실전 질문 3개
* 꼬리질문 4개
* 첨삭 코멘트 1개
* 다시 연습 필요 표시
* 답변 초안 존재

이렇게 되면 굉장히 강력해집니다.

# 9. 화면 구조 제안

## A. 이력서 보기 모드 내 히트맵 토글

상단 토글:

# 10. 어떤 문장에 질문이 연결되었는지 매핑하는 방식

이 기능의 핵심 기술 과제는
**실전 면접 질문이 이력서의 어느 부분에서 나왔는지 연결하는 것**입니다.

## 매핑 기준

### 1) 명시적 연결

사용자가 실전 면접 등록 시

* 제출 이력서 버전 연결
* 질문 단위로 직접 연결

### 2) 반자동 연결

앱이 질문 텍스트와 이력서 문장을 비교해 추천

* 기술 키워드
* 프로젝트명
* 성과 지표
* 도메인 용어
* 시간/역할 정보

### 3) 후처리 보정

사용자가 “이 질문은 이 문장에서 나온 것”이라고 수정 가능

이 세 단계가 필요합니다.

## 추가로 필요한 세분화 단계

기존에는 “이 질문은 어느 project/experience에서 나왔는가” 정도면 충분했지만,
앞으로는 “그 anchor 안의 어느 문장/구절인가”까지 내려가야 합니다.

### 추천 순서

1. anchor-level inference
2. field-level inference
   * summaryText
   * contentText
   * impactText
   * sourceText
3. sentence segmentation
4. sentence/phrase ranking
5. manual correction

즉, 질문 매핑은 다음처럼 두 단계가 됩니다.

* 어느 프로젝트인가
* 그 프로젝트 설명의 어느 문장인가

---

# 10A. 문장 매핑 규칙 제안

## 1) project-level question

질문이 전체 프로젝트 맥락을 묻는 경우:

* “이 프로젝트에서 가장 어려웠던 점은?”
* “이 프로젝트의 핵심 목표는?”

→ `targetType = block`

## 2) sentence-level question

질문이 특정 설명 문장을 검증하는 경우:

* “콘텐츠 라이프사이클 파이프라인 구조화 및 운영 안정화”가 무슨 의미였나요?

→ `targetType = sentence`

## 3) phrase-level question

질문이 특정 수치/기술/표현만 검증하는 경우:

* “운영 안정화”를 어떤 지표로 판단했나요?
* “End-to-End 파이프라인”이 구체적으로 어떤 경로였나요?

→ `targetType = phrase`

# 11. 시각 효과를 더 줄 수 있는 아이디어

## A. 호버 애니메이션

마우스를 올리면 해당 부분이 약간 더 진해지고, 연결된 질문 수가 떠오릅니다.

예:

* 질문 5
* 꼬리질문 8
* 최근 면접 2회 등장

---

## B. 질문 파동 효과

아주 빈도 높은 부분은 은은한 펄스 효과를 줄 수도 있습니다.
다만 너무 과하면 산만해지므로 선택적이어야 합니다.

추천:

* “핵심 질문 포인트 강조” 옵션일 때만 사용

---

## C. 질문 흐름 연결선

특정 이력서 문장에서 시작된 질문들이 실제 면접에서 어떤 꼬리질문으로 이어졌는지,
작은 그래프나 펼침 패널로 보여줄 수 있습니다.

예:

* 이력서 문장
  → 메인 질문
  → 검증 질문
  → 꼬리질문
  → 압박 질문

이건 2차 확장으로 매우 좋습니다.

---

## D. 히트맵 요약 카드

이력서 상단에 요약 정보를 보여줄 수 있습니다.

예:

* 가장 많이 질문된 프로젝트: API 재설계
* 가장 많이 검증된 기술: Redis
* 꼬리질문 유발 1위: 성능 개선 수치
* 답변 취약 포인트: MapStruct 검증 설계 이유

이 요약은 사용자에게 즉시 가치가 있습니다.

# 12. 이 기능이 특히 강한 포인트

이 기능은 그냥 “많이 질문된 부분 표시”에서 끝나면 아쉽습니다.
아래까지 이어져야 진짜 강해집니다.

## 1) 질문이 많이 나온 문장을 더 잘 쓰게 만들기

히트맵을 보고 해당 문장을 첨삭합니다.

## 2) 질문 카드로 바로 연습하기

짙은 부분을 눌러 해당 질문 세트로 모의면접 시작

## 3) 약점 문장만 따로 모으기

“자주 질문되고, 답변도 약했던 문장”만 추출

## 4) 회사별 차이 분석

A사에서는 이 프로젝트를 많이 물었고,
B사에서는 협업 관련 문장을 많이 물었다는 비교 가능

이런 흐름이 붙으면 정말 강력해집니다.

# 16. 최종 권장안

이 추가 기능은 아래처럼 정의하면 가장 좋습니다.

## 기능 요약

**특정 이력서 버전에 연결된 실전 면접 데이터를 분석해, 질문이 실제로 유도된 문장과 문단을 음영으로 강조하고, 질문 빈도·꼬리질문 강도·반복 검증 정도에 따라 색 농도와 시각적 표식을 달리하여 보여주는 이력서 면접 히트맵 기능**

## 핵심 가치

* 이력서에서 실제로 자주 공격받는 지점을 한눈에 파악
* 질문 카드와 연결된 실전 복기 강화
* 자주 검증되는 문장을 우선 첨삭하고 연습 가능
* 실전 경험이 쌓일수록 이력서가 더 정교한 학습 지도로 변환됨

## 추천 우선순위

1. 질문 연결
2. 빈도 기반 음영 시각화
3. 클릭 시 질문 카드 열람
4. 꼬리질문/약점 가중치 반영
5. 첨삭/시뮬레이션 연동

이 기능이 들어가면, 사용자의 이력서는 단순 제출 문서가 아니라
**실전 면접에서 어디가 검증되는지 보여주는 살아있는 학습 지도**가 됩니다.

원하시면 이어서 이 기능까지 포함한 형태로
**전체 기능 구조를 다시 정리한 통합 PRD**로 묶어드리겠습니다.
