좋습니다. 이 기능은 단순 코멘트 기능이 아니라,
**문서의 특정 문장·문단을 근거로 질문을 만들고 복기 자산으로 전환하는 “주석 기반 면접 분석 레이어”**로 설계하는 것이 가장 좋습니다.

사용자께서 올려주신 노션 화면처럼,

* **문장을 블록처럼 음영 처리해서 강조**
* **그 지점에 코멘트 작성**
* **같은 위치에 질문 카드 부착**
* 나아가 **자동으로 질문 카드 생성**

까지 되면, 이력서 첨삭, 실전 면접 복기, 모의면접 생성 기능과 매우 강하게 연결됩니다.

# 1. 기능 한 줄 정의

**이력서·채용공고·면접 스크립트·면접 복기 문서의 특정 텍스트 범위를 선택해 음영 강조하고, 해당 위치에 코멘트와 질문 카드를 수동/자동으로 연결할 수 있는 주석 기반 리뷰 기능**

---

# 2. 왜 이 기능이 중요한지

이 앱에서는 문서가 단순 읽기 자료가 아니라, 실제로 질문이 파생되는 소스입니다.

예를 들면 아래와 같습니다.

* 이력서 문장 하나가 기술 검증 질문의 출발점이 됩니다.
* 채용공고 문장 하나가 맞춤형 예상 질문의 기준이 됩니다.
* 실전 면접 스크립트 한 답변이 다음 꼬리질문의 트리거가 됩니다.
* 복기 문서의 특정 부분이 다시 연습해야 할 약점 포인트가 됩니다.

그래서 이 기능은 단순 “댓글”이 아니라,
**텍스트 → 피드백 → 질문 → 훈련**으로 이어지는 핵심 인터랙션이 됩니다.

# 5. 핵심 기능 구성

## A. 텍스트 범위 음영 강조

사용자가 문장 또는 문단 일부를 드래그하면 선택 영역이 생기고, 그 범위를 배경 음영으로 표시합니다.

### 지원 방식

* 문장 일부 inline 선택
* 한 문단 전체 선택
* 여러 문단 block 선택
* 질문-답변 구간 단위 선택

### 표시 형태

* 연노랑/연회색 계열 기본 하이라이트
* hover 시 우측에 코멘트 아이콘 표시
* 해당 블록에 코멘트/질문 카드 개수 뱃지 표시

노션처럼 “블록 전체가 은은하게 강조되는 느낌”이 중요합니다.
단순 밑줄보다 **검토 중인 포인트라는 시각적 신호**가 더 명확해야 합니다.

---

## B. 코멘트 작성

선택된 텍스트 범위에 코멘트를 남길 수 있습니다.

### 코멘트 유형

* 일반 메모
* 첨삭 의견
* 검증 포인트
* 보완 포인트
* 리마인드
* 회고 메모

### 예시

* “이 문장은 성과 수치가 없어 설득력이 약합니다.”
* “여기서 면접관이 본인 기여도를 더 물을 가능성이 높습니다.”
* “이 표현은 JD와 직접 연결되므로 강조 유지가 좋습니다.”
* “답변은 자연스럽지만 결정 이유가 빠져 있습니다.”

---

## C. 질문 카드 부착

코멘트와 별도로, 특정 텍스트 범위에 **질문 카드**를 직접 달 수 있어야 합니다.

이게 이 기능의 핵심 차별점입니다.

### 질문 카드란

특정 문장/문단에서 파생된 면접 질문 객체입니다.

즉, 단순 메모가 아니라 실제 훈련 가능 자산입니다.

### 예시

선택한 문장:

* “Redis 캐시를 적용해 응답 시간을 4배 개선했습니다.”

여기에 붙는 질문 카드:

* “어떤 데이터를 캐시했습니까?”
* “캐시 무효화 전략은 무엇이었습니까?”
* “왜 DB 튜닝보다 캐시를 먼저 선택했습니까?”
* “응답 시간이 4배 개선되었다는 수치는 어떻게 측정했습니까?”

### 질문 카드 속성

* 질문 본문
* 질문 유형
* 난이도
* 출처 텍스트
* 연결 문서
* 수동 생성 / 자동 생성 여부
* 저장 상태
* 모의면접 포함 여부

---

# 6. 질문 카드의 두 가지 생성 방식

## 1) 수동 생성

사용자가 직접 선택 텍스트에 질문을 붙이는 방식입니다.

### 흐름

텍스트 선택 → “질문 카드 추가” → 질문 입력 → 저장

### 장점

* 사용자가 직접 중요한 포인트를 학습 포인트로 지정 가능
* AI 없이도 바로 활용 가능
* 개인화 강함

---

## 2) 자동 생성

선택한 텍스트를 기반으로 앱이 질문 카드를 추천하거나 자동 생성하는 방식입니다.

이 부분이 매우 중요합니다.

### 자동 생성 트리거 예시

* 사용자가 텍스트를 하이라이트했을 때
* 첨삭 결과에서 “검증 필요” 문장이 탐지되었을 때
* 실전 면접 전사에서 새로운 기술 키워드가 발견되었을 때
* JD와 연결된 핵심 요구사항 문장이 선택되었을 때
* 사용자가 “자동 질문 생성” 버튼을 눌렀을 때

### 자동 생성 질문 유형

* 사실 검증 질문
* 기술 깊이 질문
* 꼬리질문
* 수치 검증 질문
* 본인 기여도 질문
* 의사결정 질문
* 리스크/트레이드오프 질문
* 협업/갈등 질문
* 행동형 질문

---

# 7. 자동 질문 카드 생성 규칙

이 기능은 단순 문장 패러프레이즈가 아니라,
**텍스트의 성격에 따라 질문 타입을 다르게 생성**해야 합니다.

## A. 이력서 문장에 대한 자동 질문

예시 문장:

* “대용량 트래픽 환경에서 API 성능 최적화를 수행했습니다.”

자동 생성 카드:

* 어떤 병목이 가장 컸습니까?
* 어떤 지표로 병목을 확인했습니까?
* 구체적으로 어떤 최적화를 적용했습니까?
* 적용 전후 수치는 어떻게 달라졌습니까?
* 이 과정에서 본인이 맡은 역할은 무엇이었습니까?

---

## B. 채용공고 문장에 대한 자동 질문

예시 문장:

* “대규모 분산 시스템 운영 경험이 있는 분을 찾습니다.”

자동 생성 카드:

* 분산 환경에서 장애를 처리한 경험이 있습니까?
* 데이터 정합성과 가용성 간 트레이드오프를 설명해주시겠습니까?
* 운영 중 병목이나 장애를 어떻게 관찰하고 대응했습니까?

---

## C. 실전 면접 답변에 대한 자동 질문

예시 답변:

* “캐시를 활용해 성능을 개선했습니다.”

자동 생성 카드:

* 어떤 데이터를 캐시했습니까?
* 캐시 적중률은 확인했습니까?
* 정합성 문제는 없었습니까?
* 캐시 만료 정책은 어떻게 잡았습니까?

---

## D. 첨삭 코멘트 기반 자동 질문

예시 코멘트:

* “성과 수치가 부족합니다.”

자동 생성 카드:

* 성과를 수치로 설명해주실 수 있습니까?
* 개선 효과를 어떤 기준으로 측정했습니까?

---

# 8. 질문 카드의 역할

질문 카드는 그냥 달아두고 끝나는 것이 아니라, 기존 DB와 연결되어야 합니다.

## 질문 카드를 통해 할 수 있어야 하는 것

* 질문은행에 저장
* 모의면접 세션에 포함
* 실전 면접 복기에서 재연습 대상으로 지정
* 태그 기준으로 묶기
* 답변 초안 작성
* 꼬리질문 자동 생성의 seed로 사용
* 유사 질문 추천

즉, 질문 카드는 주석 UI의 부산물이 아니라
**앱의 핵심 데이터 객체**여야 합니다.

이 구조는 이력서 첨삭 화면이나 실전 면접 복기 화면에서 특히 유용합니다.

# 10. 이 앱에 맞는 핵심 사용 시나리오

## 시나리오 1: 이력서 첨삭 중

사용자가 이력서 문장을 하이라이트합니다.

문장:

* “UI 의존 API를 RESTful 도메인 API로 재설계했습니다.”

앱이 자동으로 질문을 추천합니다.

* 왜 기존 UI 의존 API 구조가 문제였습니까?
* 도메인 중심으로 재설계하면서 어떤 책임 분리가 이루어졌습니까?
* 성능은 어떻게 측정했고 어느 정도 개선되었습니까?
* 프론트엔드와의 결합도는 어떻게 낮아졌습니까?

사용자는 그중 2개를 질문 카드로 저장합니다.

---

## 시나리오 2: 실전 면접 복기 중

전사본에서 답변 일부를 하이라이트합니다.

답변:

* “MapStruct 매핑 단계에서 검증을 넣었습니다.”

앱이 자동 질문을 생성합니다.

* 왜 DTO 단계가 아니라 매핑 단계에서 검증했습니까?
* Bean Validation만으로는 부족했던 이유는 무엇입니까?
* 매핑 단계 검증이 도메인 안정성에 어떤 도움이 되었습니까?

사용자는 코멘트를 남깁니다.

* “여기서 상태 조합 검증을 더 구체적으로 설명했어야 함”

이후 이 질문 카드는 “다시 연습” 목록으로 들어갑니다.

---

## 시나리오 3: 채용공고 분석 중

JD의 특정 문장을 하이라이트합니다.

문장:

* “운영 환경에서의 장애 대응 경험”

자동 카드:

* 운영 중 장애를 탐지한 방식은 무엇이었습니까?
* 가장 어려웠던 장애와 대응 과정을 설명해주시겠습니까?
* 재발 방지를 위해 어떤 조치를 했습니까?

이 질문들은 해당 회사 맞춤 모의면접에 포함됩니다.

---

# 11. 자동 질문 생성 수준 설계

자동 생성은 세 단계로 구분하는 것이 좋습니다.

## 1단계: 추천 질문

선택 영역에 대해 3~5개의 질문 후보를 보여줍니다.
사용자가 저장 여부를 결정합니다.

## 2단계: 자동 부착

중요도가 높은 경우 질문 카드를 자동으로 생성해 붙입니다.
예:

* 성과 수치 문장
* 기술 검증 문장
* JD 핵심 요구사항

## 3단계: 자동 꼬리질문 확장

이미 생성된 질문 카드에 대해 꼬리질문까지 자동으로 묶습니다.

예:

* 메인 질문 1개
* 검증 꼬리질문 2개
* 반례/트레이드오프 질문 1개

---

# 12. 질문 카드 유형 체계

질문 카드는 반드시 유형화해야 합니다.

## 추천 타입

* `verification` 검증형
* `deep_dive` 기술 심화형
* `metric` 수치 검증형
* `ownership` 본인 기여도 확인형
* `tradeoff` 대안/트레이드오프형
* `behavioral` 행동 경험형
* `follow_up` 꼬리질문형
* `resume_based` 이력서 기반
* `jd_based` 공고 기반
* `transcript_based` 실전 답변 기반

이렇게 해두면

* 질문 필터링
* 시뮬레이션 출제 규칙
* 약점 분석
  에 모두 활용할 수 있습니다.

# 14. 텍스트 앵커링 설계

이 기능은 구현에서 여기서 가장 중요합니다.

문서가 수정되면 단순 start/end offset만으로는 주석 위치가 깨질 수 있습니다.
그래서 **복합 앵커 방식**이 필요합니다.

## 권장 방식

* blockId
* startOffset / endOffset
* anchorQuote
* prefix / suffix context

즉,
“몇 번째 문자”만 저장하지 말고
“선택된 원문 일부와 주변 문맥”도 같이 저장해야 합니다.

그래야 문서가 조금 수정돼도 주석을 다시 매칭할 수 있습니다.

## 재매칭 우선순위 예시

1. blockId 일치
2. anchorQuote exact match
3. prefix/suffix 포함 유사 match
4. fallback으로 orphan 상태 처리

## orphan 상태

원문이 크게 수정되어 앵커를 잃은 경우

* “연결이 약해진 주석”으로 표시
* 사용자가 수동 재연결 가능

이 부분은 실제 품질에 매우 중요합니다.

# 15. 기존 기능과의 연결 방식

## A. 이력서 첨삭 기능과 연결

하이라이트한 문장에

* 첨삭 코멘트
* 예상 질문 카드
* 실제 면접에서 나왔던 유사 질문

을 함께 붙일 수 있습니다.

즉,
**문장 첨삭과 면접 대비가 한 화면에서 이어집니다.**

---

## B. 실전 면접 복기 기능과 연결

전사/정제본의 특정 답변에

* 당시 부족했던 점 코멘트
* 다시 연습할 질문 카드
* 다음에 받을 꼬리질문 카드

를 직접 달 수 있습니다.

즉,
**복기 문서가 곧 훈련 세트가 됩니다.**

---

## C. 시뮬레이션 기능과 연결

저장된 질문 카드는 아래처럼 활용할 수 있습니다.

* 특정 문서 기반 질문만 출제
* 특정 하이라이트 영역에서 생성된 질문만 출제
* unresolved 코멘트가 달린 영역 우선 출제
* 자동 생성 질문과 수동 생성 질문을 섞어 출제

# 17. 자동 생성 UX 추천

자동 생성은 “무조건 붙이기”보다 아래 방식이 좋습니다.

## 권장 방식

* 기본은 추천
* 중요도가 높은 경우만 자동 부착
* 자동 부착된 카드는 “AI 생성” 뱃지 표시
* 사용자가 승인/삭제 가능

## 자동 생성 우선 대상

* 기술 용어가 포함된 성과 문장
* 과장 가능성이 있는 문장
* JD 핵심 요구사항 문장
* 실전 면접에서 추가 검증이 붙었던 문장
* 답변이 추상적이었던 전사 구간

# 20. 최종 권장안

이 기능은 이렇게 정의하는 것이 가장 좋습니다.

## 기능 요약

**문서 내 특정 텍스트를 음영 강조하여 코멘트와 질문 카드를 연결하고, 선택 문맥을 바탕으로 면접 질문을 수동 또는 자동 생성하여 저장·복기·시뮬레이션까지 이어지게 하는 기능**

## 핵심 가치

* 문서 검토와 면접 대비를 하나로 연결
* 중요한 문장을 질문 자산으로 변환
* 실전 복기를 구조화
* 자동 질문 생성으로 학습 효율 향상

## 추천 우선순위

1. 하이라이트 + 코멘트
2. 질문 카드 수동 추가
3. 자동 질문 추천
4. 기존 질문 DB/시뮬레이션 연결
5. 자동 꼬리질문 확장

이 기능이 들어가면, 사용자는 더 이상 문서를 “읽기만” 하지 않고
**문서 위에서 바로 질문을 만들고, 생각을 남기고, 연습 자산으로 전환**할 수 있게 됩니다.

원하시면 다음 단계로 이어서 이 기능까지 포함한 형태로
**전체 제품 정보구조 재정리**,
또는 **백엔드/프론트엔드 구현용 상세 PRD**로 구체화해드리겠습니다.
