좋습니다. 이 기능은 매우 강력합니다.
잘 설계하면 이 앱이 단순 연습용 면접 앱이 아니라, **실전 면접 기록을 학습 자산으로 축적하고, 그 데이터를 바탕으로 같은 면접관 스타일을 재현해 훈련하는 시스템**으로 발전할 수 있습니다.

핵심은 아래 4단계를 하나의 흐름으로 묶는 것입니다.

1. **실제 면접 녹음 업로드**
2. **스크립트 생성 및 문맥 기반 정제**
3. **질문/답변 구조화 및 DB 저장**
4. **복기 + 동일 스타일 재시뮬레이션**

이 기능은 기존에 기획한

* 이력서 첨삭
* 채용공고 기반 맞춤화
* 면접 질문 생성
* 꼬리질문 시뮬레이션

과 매우 자연스럽게 연결됩니다.

# 2. 이 기능이 해결하는 핵심 문제

실제 면접을 보고 나면 사용자는 보통 아래 상태가 됩니다.

* 무슨 질문이 나왔는지 일부만 기억납니다.
* 내 답변이 실제로 어땠는지 정확히 모릅니다.
* 어디서 버벅였는지 복기가 어렵습니다.
* 면접관이 어떤 스타일이었는지 감으로만 기억합니다.
* 같은 회사/비슷한 면접관을 다시 만나면 대비하기 어렵습니다.
* 이전 면접에서 나온 질문 패턴을 다음 면접 준비에 연결하지 못합니다.

이 기능은 이 문제를 해결합니다.

### 핵심 가치

* 실제 면접을 **구조화된 데이터**로 남깁니다.
* 질문/답변을 **다시 꺼내볼 수 있게** 합니다.
* 면접관의 **질문 패턴과 주제 성향**을 추출합니다.
* 사용자의 답변 약점을 분석합니다.
* 동일한 스타일의 면접을 **다시 시뮬레이션**할 수 있게 합니다.
* 같은 질문이라도 답변에 따라 **다른 꼬리질문 흐름**을 만듭니다.

즉, 이 기능은
**실전 경험을 소비하지 않고 축적하는 장치**입니다.

# 3. 제품 한 줄 정의

**실제 면접 녹음 파일을 업로드하면 스크립트를 생성·정제하고, 질문/답변을 구조화하여 저장한 뒤, 해당 면접의 질문 패턴·주제·꼬리질문 성향을 학습해 복기 및 재시뮬레이션까지 제공하는 기능**

# 4. 핵심 사용자 시나리오

## 시나리오 A: 면접 직후 복기

1. 사용자가 실제 면접 녹음 파일을 업로드합니다.
2. 앱이 자동으로 스크립트를 생성합니다.
3. 문맥상 이상한 단어, 오인식 표현, 끊긴 문장을 보정합니다.
4. 질문/답변 단위로 분리합니다.
5. 각 질문에 대해 사용자의 답변 품질과 놓친 포인트를 분석합니다.
6. 면접 전체 흐름과 주요 질문 주제를 저장합니다.
7. 사용자는 나중에 면접 단위 또는 질문 단위로 다시 열람합니다.

## 시나리오 B: 같은 회사/유사 면접 재연습

1. 사용자가 예전에 본 실전 면접 기록을 선택합니다.
2. 앱이 면접관의 질문 스타일, 주제 우선순위, 압박 정도, 꼬리질문 습관을 분석합니다.
3. 새 시뮬레이션 세션을 시작합니다.
4. 기존 질문자와 유사한 패턴으로 질문을 시작합니다.
5. 사용자의 새 답변 내용에 따라 꼬리질문은 동적으로 바뀝니다.
6. 기존 실전 면접과 이번 답변을 비교해 차이를 분석합니다.

## 시나리오 C: 제출한 이력서가 있는 경우

1. 사용자가 실제 지원 시 제출했던 이력서를 연결합니다.
2. 앱은 질문이 이력서 어느 섹션에서 유도되었는지 추정합니다.
3. “이 질문은 해당 프로젝트 문장에서 파생된 질문”처럼 연결해서 보여줍니다.
4. 시뮬레이션 시에도 같은 이력서 포인트를 중심으로 질문이 나옵니다.

## 시나리오 D: 제출 이력서가 없는 경우

1. 앱은 실전 면접 녹취에서 주제, 기술 키워드, 검증 포인트를 추출합니다.
2. 사용자의 답변 내용과 면접 흐름을 바탕으로 가상 면접관 프로필을 생성합니다.
3. 이 프로필을 기반으로 비슷한 주제의 질문을 재구성합니다.
4. 사용자의 새 답변에 따라 실시간으로 다른 꼬리질문을 이어갑니다.

# 5. 기능 구성의 핵심 구조

이 기능은 아래 5개 모듈로 설계하는 것이 가장 적절합니다.

## A. 녹음 업로드 및 스크립트 생성

입력:

* mp3
* m4a
* wav
* mp4에서 오디오 추출 가능

처리:

* 음성 인식으로 원문 전사
* 시간축 단위 세그먼트 생성
* 화자 구분 가능하면 interviewer / candidate 분리
* 화자 분리가 애매하면 문맥 분류로 질문자/응답자 추정

출력:

* 원본 스크립트
* 문장 단위 세그먼트
* 타임스탬프
* 화자 분류 결과
* 신뢰도 점수

여기서 중요한 점은
**전사 결과를 무조건 최종본으로 쓰지 말고, 원문과 정제본을 모두 보관해야 한다는 점**입니다.

---

## B. 문맥 기반 스크립트 정제

사용자가 원하신 “맥락상 이상한 단어를 더 다듬는 기능”은 별도 단계로 분리하는 것이 좋습니다.

### 왜 필요한가

실제 면접 녹음 전사는 아래 문제가 많습니다.

* 기술 용어 오인식
* 회사명/서비스명 오인식
* 한국어+영어 혼합 표현 깨짐
* 문장 끊김
* “어…”, “그…” 같은 필러 노이즈 과다
* 질문과 답변 경계 불명확

### 정제 기능이 해야 할 일

* 문맥상 부자연스러운 단어 교정
* 기술 용어 사전 기반 보정
* 인명/회사명/서비스명 보정
* 조사와 문장 흐름 보정
* 지나친 필러 제거
* 질문문/답변문 문장화
* 너무 공격적인 자동 재작성은 금지

### 정제 결과는 3단계가 좋습니다

1. **원본 전사**
2. **자동 정제본**
3. **사용자 수정 확정본**

이 구조가 필요한 이유는,
나중에 사용자가 “이 단어는 잘못 고쳐졌다”라고 느낄 수 있기 때문입니다.

### 예시

원본 전사:

* “레디스 캐쉬를 써가지고 응답시간이 한 4배 정도 줄었고요”

정제본:

* “Redis 캐시를 적용해 응답 시간을 약 4배 단축했습니다.”

원본이 가진 의미는 유지하되,
면접 복기와 학습에 적합한 문장으로 다듬는 수준이 적절합니다.

## C. 질문/답변 구조화 추출

이 모듈이 DB 저장의 핵심입니다.

### 추출해야 하는 것

* 질문 시작/종료 시점
* 답변 시작/종료 시점
* 질문 유형
* 답변 유형
* 연결된 꼬리질문 여부
* 이전 질문과의 관계
* 질문 주제
* 관련 이력서 항목 추정
* 관련 JD 항목 추정
* 답변 품질/안정성

### 질문 유형 예시

* 자기소개
* 이력서 검증
* 프로젝트 설명
* 기술 깊이
* 문제 해결
* 성과 검증
* 협업/갈등
* 동기/이직 사유
* 상황 질문
* 압박 질문
* 마무리 질문

### 질문 단위 저장 예시

질문:

* “Redis를 사용하셨다고 했는데 캐시 무효화는 어떻게 하셨나요?”

답변:

* 사용자가 말한 실제 답변

메타데이터:

* topic: caching / redis / consistency
* intent: 기술 검증
* derivedFrom: resume.project[2]
* followUpTo: previous question id
* interviewerStyleTag: depth_probe

이렇게 저장되어야 나중에 질문 단위 검색과 시뮬레이션이 가능해집니다.

## D. 복기 분석

이 기능의 절반은 저장이고, 나머지 절반은 복기입니다.

### 면접 단위 분석

* 전체 질문 수
* 질문 카테고리 분포
* 기술 질문 비중
* 꼬리질문 빈도
* 특정 주제 반복 여부
* 압박 강도
* 답변 길이
* 답변 중단/버벅임 지점
* 답변 불확실 표현 빈도
* 약했던 답변 구간

### 질문 단위 분석

* 질문 의도
* 왜 이 질문이 나왔는지
* 사용자의 답변에서 부족했던 점
* 더 좋았을 답변 방향
* 예상되는 추가 꼬리질문
* 이력서/JD 연결 포인트

### 예시 피드백

* 이 질문은 단순 기술 확인이 아니라 “본인이 직접 설계했는지”를 검증하는 질문입니다.
* 답변에서 캐시 전략은 언급했지만 무효화 시점과 데이터 정합성 설명이 부족했습니다.
* 이 답변은 사실 중심 서술은 좋았지만, 수치와 트레이드오프 설명이 빠졌습니다.
* 여기서 면접관은 이전 답변의 “성능 개선” 표현을 근거로 더 깊게 파고든 것입니다.

이런 분석이 있어야 복기의 가치가 생깁니다.

## E. 동일 스타일 재시뮬레이션

이 부분이 가장 차별화됩니다.

사용자가 말씀하신 핵심은 정확히 이것입니다.

> 이미 진행했던 면접을 새로 진행한다고 하면, 기존 질문자와 비슷한 패턴과 주제로 질문하되, 이번 답변에 따라 다른 꼬리질문으로 이어져야 합니다.

이것은 단순 질문 재생이 아니라
**면접관 패턴 모델링 + 동적 추론형 시뮬레이션**입니다.

# 6. 재시뮬레이션 기능 설계

## 1) 면접관 프로필 추출

실전 면접 기록으로부터 “가상 면접관 프로필”을 만듭니다.

### 추출 항목

* 질문 톤
* 질문 길이
* 공격성/압박 정도
* 기술 깊이 선호
* 꼬리질문 빈도
* 꼬리질문 방식
* 반복 검증 성향
* 관심 주제
* 질문 순서 패턴
* 답변 중 어떤 표현에 반응하는지

### 예시 프로필

* 경력 검증형
* 성능/운영 중심
* 답변 중 수치 표현이 나오면 바로 검증
* 본인 기여도와 의사결정 근거를 집요하게 확인
* 추상적 표현에 대해 즉시 구체화 질문

이 프로필을 바탕으로 시뮬레이터가 동작해야 합니다.

---

## 2) 질문 재생성 방식

새 세션에서는 기존 질문을 그대로 반복할 수도 있고, 변형된 형태로 물을 수도 있습니다.

### 질문 생성 기준

* 기존 면접관이 선호했던 주제
* 기존 질문 순서
* 연결된 이력서 포인트
* 연결된 JD 요구사항
* 사용자의 새 답변 내용
* 이전 실전 면접에서 약했던 포인트

### 질문 생성 모드

#### a. 원본 재현 모드

가능한 한 실제 면접과 유사하게 질문합니다.

#### b. 유사 패턴 모드

주제와 스타일은 유지하되 문장은 새롭게 생성합니다.

#### c. 변형 압박 모드

기존 면접관의 성향을 유지하되 더 깊게 파고듭니다.

이 세 가지 모드는 모두 유용합니다.

---

## 3) 꼬리질문 동적 생성

여기가 핵심입니다.

앱은 사용자의 새 답변을 듣고 아래를 분석해야 합니다.

* 모호한 표현
* 정량 근거 부족
* 기술적 허점
* 본인 기여도 불명확
* 결정 이유 미흡
* 리스크/대안 설명 부족
* 답변 회피
* 이전 답변과 모순

그리고 기존 면접관의 스타일에 맞춰 꼬리질문을 생성합니다.

### 예시

기존 질문:

* “트래픽 증가 시 어떤 대응을 하셨나요?”

이번 답변:

* “캐시를 적극 활용해서 해결했습니다.”

기존 면접관 스타일이 검증형이면:

* “어떤 데이터를 캐시했는지 구체적으로 말씀해주시겠습니까?”
* “캐시 적중률은 확인하셨습니까?”
* “정합성 문제는 어떻게 관리하셨습니까?”
* “왜 DB 튜닝보다 캐시를 우선 선택하셨습니까?”

즉, 질문의 **주제와 압박 방식은 유지**하면서,
세부 꼬리질문은 **이번 답변에 따라 달라지는 구조**가 맞습니다.

# 7. 이력서가 있는 경우와 없는 경우의 처리

## 이력서가 있는 경우

이력서가 있으면 가장 좋은 상태입니다.

### 활용 방식

* 질문이 어떤 이력서 문장에서 파생되었는지 연결
* 프로젝트별 질문맵 생성
* 면접관이 집요하게 파고든 이력서 항목 식별
* 같은 이력서 기반 재시뮬레이션 가능

### 예시

이력서 문장:

* “Redis를 활용한 성능 개선 경험”

실전 질문:

* “어떤 병목이었습니까?”
* “무효화 전략은 무엇이었습니까?”
* “얼마나 개선되었습니까?”

이렇게 연결되면, 이력서 첨삭 기능과도 강하게 연결됩니다.

## 이력서가 없는 경우

이 경우에도 충분히 동작해야 합니다.

### 대체 정보원

* 면접 녹취 텍스트
* 질문 주제 분포
* 답변에서 언급한 경험
* 채용공고가 있다면 JD
* 사용자 프로필/직무 설정
* 기존 면접 이력 전체

### 동작 방식

앱이 면접 중 자주 나온 키워드와 검증 포인트를 추출합니다.
예를 들면

* 분산처리
* 캐싱
* 장애 대응
* 협업 갈등
* 성능 최적화
* 이직 사유

이런 주제를 기반으로 면접관 스타일을 재구성합니다.

즉, 이력서가 없어도
**실전 면접 대화 자체가 새로운 학습 소스**가 됩니다.

# 12. 재시뮬레이션 엔진 설계

이 엔진은 다음 세 가지를 입력으로 사용합니다.

### 입력 1: 소스 면접

* 질문 순서
* 주제 분포
* 꼬리질문 패턴
* 압박 강도

### 입력 2: 사용자 맥락

* 제출 이력서
* 채용공고
* 이전 실전 답변
* 현재 연습 목표

### 입력 3: 실시간 새 답변

* 내용
* 구체성
* 수치
* 불확실성
* 회피 여부
* 기술적 허점

이 세 가지를 조합해서 다음 질문을 결정합니다.

### 질문 선택 로직 개념

1. 기존 면접의 주제 우선순위를 따릅니다.
2. 현재 답변이 충분하면 다음 주제로 넘어갑니다.
3. 현재 답변이 약하면 같은 스타일의 꼬리질문을 생성합니다.
4. 이전 실전에서 유독 집요했던 포인트는 우선 재검증합니다.
5. 사용자가 이번에는 다르게 답하면, 그 답에 맞는 새로운 검증 질문을 만듭니다.

즉,
**같은 면접관이지만 매번 완전히 동일하지는 않은 면접**을 만들어야 합니다.

# 13. 기존 기능과의 연결

이 기능은 기존 기능과 붙을 때 훨씬 강해집니다.

## 이력서 첨삭과 연결

* 실제 면접에서 파고든 이력서 문장을 식별합니다.
* 이력서의 어떤 표현이 검증 질문을 유발했는지 보여줍니다.
* 해당 문장을 더 명확하게 바꾸는 첨삭으로 연결할 수 있습니다.

## 채용공고 맞춤 기능과 연결

* 공고에서 중요하게 본 역량이 실전 면접에 실제로 얼마나 반영됐는지 보여줍니다.
* 다음 지원 시 어떤 포인트를 더 강조해야 하는지 도출할 수 있습니다.

## 면접 질문 생성 기능과 연결

* 실전 질문 DB가 쌓일수록 AI가 더 현실적인 질문을 만들 수 있습니다.
* 일반 질문은행보다 훨씬 강한 자산이 됩니다.

## 꼬리질문 시각화와 연결

* 실전 면접에서 실제로 이어진 꼬리질문 흐름을 그래프로 보여줄 수 있습니다.
* 사용자는 어느 답변이 어떤 추가 질문을 유발했는지 볼 수 있습니다.

# 17. 구현 시 중요한 주의점

## 1) 전사 오류를 과감히 정제하되, 과도하게 미화하면 안 됩니다

사용자의 실제 답변 흐름이 사라지면 복기 가치가 줄어듭니다.
따라서 “문맥상 이상한 단어를 바로잡는 수준”이 우선이며,
답변을 지나치게 예쁘게 다시 쓰는 것은 별도 보기로 분리하는 것이 좋습니다.

## 2) 질문/답변 경계는 사용자가 수정 가능해야 합니다

실전 녹취는 완벽히 자동화하기 어렵습니다.

## 3) 실제 면접 데이터는 민감합니다

회사명, 프로젝트명, 개인정보, 기밀 내용이 들어갈 수 있으므로

* 비공개 저장
* 삭제 기능
* 민감정보 마스킹 옵션
  이 중요합니다.

## 4) 시뮬레이션은 “같은 질문 반복기”가 되면 안 됩니다

핵심은 면접관 스타일 재현 + 답변 기반 동적 꼬리질문입니다.
