좋습니다.
현재 프로젝트 방향을 기준으로 보면, 이력서 편집 화면은 **단순 폼 기반 편집기보다 노션식 블록 에디터에 가까운 마크다운 기반 편집기**로 가는 것이 훨씬 적합합니다.

다만 그대로 “노션 복제”로 가기보다, 이 앱 목적에 맞게 아래처럼 정의하는 것이 좋습니다.

> **이력서 편집 화면은 마크다운을 기반으로 하되, 블록 단위 편집·인라인 포맷·코멘트·질문 카드·히트맵·PDF 출력을 모두 지원하는 이력서 워크스페이스로 설계한다.**

즉, 문서를 쓰는 곳이면서 동시에

* 첨삭하는 곳
* 질문이 파생되는 곳
* 실전 면접 데이터가 덧씌워지는 곳
* 제출본을 만드는 곳

이어야 합니다.

# 현재 프로젝트 기준 백엔드 해석

현재 백엔드는 이미 다음 전제를 가지고 있습니다.

* `resume_versions`는 immutable record입니다.
* parsed resume snapshot은 별도 테이블에 저장됩니다.
* 실전 면접 heatmap은 additive read model입니다.
* tailored resume analysis도 source version 위에 별도 레이어로 저장됩니다.

그래서 이 문서의 “resume editor”는 source version 자체를 수정하는 방식이 아니라,
**immutable resume version 위에 editor workspace draft를 얹는 구조**로 해석하는 것이 현재 프로젝트와 가장 잘 맞습니다.

현재 구현된 백엔드 slice는 다음입니다.

* `GET /api/resume-versions/{versionId}/editor`
* `PUT /api/resume-versions/{versionId}/editor/document`
* `POST /api/resume-versions/{versionId}/editor/import-markdown`
* `POST /api/resume-versions/{versionId}/editor/comments`
* `PATCH /api/resume-versions/{versionId}/editor/comments/{commentId}`
* `POST /api/resume-versions/{versionId}/editor/comments/{commentId}/replies`
* `POST /api/resume-versions/{versionId}/editor/presence`
* `POST /api/resume-versions/{versionId}/editor/question-cards`
* `PATCH /api/resume-versions/{versionId}/editor/question-cards/{cardId}`
* `POST /api/resume-versions/{versionId}/editor/auto-question-suggestions`
* `POST /api/resume-versions/{versionId}/editor/rewrite-suggestions`
* `GET /api/resume-versions/{versionId}/editor/print-preview`
* `GET /api/resume-versions/{versionId}/editor/revisions`
* `GET /api/resume-versions/{versionId}/editor/revisions/{revisionId}`
* `GET /api/resume-versions/{versionId}/editor/tracked-changes`
* `POST /api/resume-versions/{versionId}/editor/merge-preview`

즉 현재는 프론트가 바로 붙일 수 있는 **workspace contract + annotation persistence + markdown import + print preview + deterministic suggestion API + optimistic revision history + lightweight presence API + tracked changes + merge preview API**까지 구현된 상태로 보는 것이 맞습니다.

# 1. 왜 노션식 마크다운 편집기가 맞는가

현재 기획된 기능들을 보면 이력서 편집 화면은 단순 입력 UI로는 부족합니다.

이미 들어가야 하는 기능이 많습니다.

* 문장 단위 첨삭
* 하이라이트
* 코멘트
* 질문 카드
* 자동 질문 추천
* 실전 면접 히트맵
* 회사별 버전 관리
* PDF 출력
* 이력서 문장과 실전 질문 연결

이 기능들은 전부 **문서 위에서 직접 작동**해야 가치가 큽니다.
그래서 이력서 편집 화면은 그냥 form editor가 아니라 **document-centered editor**여야 합니다.

노션식 접근이 좋은 이유는 다음과 같습니다.

* 블록 단위 조작이 쉽습니다.
* 특정 문장/문단을 선택해 주석 달기 좋습니다.
* 섹션 순서 변경이 자연스럽습니다.
* 마크다운 기반이면 데이터 portability가 좋습니다.
* PDF 출력용 구조화에도 유리합니다.

---

# 2. 이 화면의 역할 재정의

이력서 편집 화면은 단순 “편집기”가 아니라 아래 4가지 역할을 동시에 가져야 합니다.

## 1) 작성 도구

사용자가 이력서를 자유롭게 작성/수정할 수 있어야 합니다.

## 2) 리뷰 도구

AI 첨삭, 코멘트, 주석, 질문 카드가 같은 문서 위에서 작동해야 합니다.

## 3) 학습 도구

실전 면접에서 질문이 많이 나온 부분을 히트맵으로 보여줘야 합니다.

## 4) 출력 도구

최종 제출용 구조와 PDF 품질을 보장해야 합니다.

즉, 이 편집 화면은
**Resume Editor**가 아니라 **Resume Workspace**에 가깝습니다.

---

# 3. 핵심 방향

## 최종 방향 제안

**블록 기반 리치 마크다운 에디터 + 인라인 주석/질문 카드 + 이력서 전용 구조 제약 + 제출용 렌더링 분리**

이 네 가지를 같이 가져가야 합니다.

중요한 점은 “자유도”와 “제출 안정성”을 동시에 가져가야 한다는 것입니다.
그래서 완전 자유 문서 편집기보다, **이력서 문서에 맞게 제한된 블록 타입을 가진 에디터**가 적절합니다.

---

# 4. 추천 편집 모델

## 기본 모델

예를 들면 한 문서는 아래처럼 구성됩니다.

* Header block
* Summary block
* Experience section

    * Company block
    * Role block
    * Bullet blocks
* Project section
* Skills section
* Education section

각 블록 안에서는 마크다운 스타일의 인라인 포맷을 허용합니다.

예:

* 굵게
* 기울임
* 인라인 코드
* 링크
* 하이라이트

이 구조가 필요한 이유는,
이력서는 일반 노트와 달리 **구조적 문서**이기 때문입니다.

즉, 내부는 리치 문서 모델이지만, 사용자는 마크다운처럼 입력할 수 있어야 합니다.

* underline
* strike
* inline code
* highlight

# 6. 현재 프로젝트에 꼭 필요한 기능

이 부분이 핵심입니다.
그냥 문서 편집기가 아니라, 현재 기획 중인 제품에 맞게 아래 기능이 편집기 안에 직접 들어와야 합니다.

---

## A. 텍스트 선택 후 코멘트

노션처럼 특정 문장 범위를 선택하면 코멘트를 달 수 있어야 합니다.

예:

* “이 문장은 성과 수치가 빠져 있습니다.”
* “이 표현은 본인 기여도가 모호합니다.”
* “이 부분은 실제 면접에서 자주 질문된 포인트입니다.”

이 기능은 필수입니다.

### 요구사항

* 문장 일부 선택 가능
* 문단 전체 선택 가능
* 우측 사이드 패널에 코멘트 스레드 표시
* 해결됨 / 미해결 상태 표시
* 해당 텍스트에 코멘트 개수 뱃지 표시

---

## B. 텍스트 선택 후 질문 카드 추가

이 앱의 차별점이므로 꼭 들어가야 합니다.

예:
선택 문장:

* “Redis 캐시를 적용해 응답 시간을 4배 개선했습니다.”

달리는 질문 카드:

* 어떤 데이터를 캐시했습니까?
* 캐시 무효화 전략은 무엇입니까?
* 4배 개선은 어떻게 측정했습니까?

### 액션 메뉴 예시

텍스트 선택 시 툴바:

* 하이라이트
* 코멘트
* 질문 카드 추가
* 자동 질문 생성
* 태그 추가

이 흐름이 매우 자연스럽습니다.

---

## C. 자동 질문 추천

편집 화면에서 중요한 문장을 선택하거나, AI가 검증 포인트를 탐지했을 때 질문을 추천해야 합니다.

예:

* 성과 수치 문장
* 기술 용어가 들어간 문장
* 과장 가능성이 있는 문장
* JD와 강하게 연결되는 문장

이때 편집기 우측 패널에서 아래처럼 보여줄 수 있습니다.

* 추천 질문 3개
* 저장 버튼
* 시뮬레이션 포함 토글
* 꼬리질문 확장 버튼

---

## D. 이력서 히트맵 오버레이

특정 이력서 버전에서 실전 면접 데이터가 쌓이면 편집기 자체가 히트맵 뷰를 지원해야 합니다.

즉, 편집 화면 안에서 아래 뷰 전환이 가능해야 합니다.

* 일반 편집 보기
* 코멘트 보기
* 질문 카드 보기
* 히트맵 보기

히트맵 보기에서는

* 자주 질문된 문장 = 더 진한 음영
* 꼬리질문 많음 = 추가 아이콘
* 답변 취약 포인트 = 경고 뱃지

이렇게 보여주는 것이 좋습니다.

---

## E. 첨삭 제안 인라인 반영

편집기 우측 패널 또는 인라인 suggestion 형태로 AI 첨삭을 제공해야 합니다.

예:
원문:

* “서비스 성능 개선을 담당했습니다.”

제안:

* “주요 API 병목을 분석하고 Redis 캐시 및 쿼리 최적화를 적용해 평균 응답 시간을 1.8초에서 450ms로 개선했습니다.”

사용자는

* 제안 적용
* 비교 보기
* 일부만 적용
* 무시

를 할 수 있어야 합니다.

# 8. 노션처럼 지원하되, 현재 프로젝트에 맞게 제한해야 할 부분

* 히트맵/질문 연결 정확도가 떨어질 수 있습니다.

그래서 다음처럼 가는 것이 좋습니다.

## 허용

* 블록 기반 작성
* 인라인 포맷
* 댓글/질문 카드
* 블록 순서 변경
* 섹션 이동
* 마크다운 입력
* 붙여넣기 파싱

* 복잡한 다단 컬럼 자유 배치
* 임의의 임베드 남용
* 지나치게 자유로운 색상/폰트 변경
* 테이블 중심 자유 레이아웃
* 레이아웃이 깨질 수 있는 스타일 조합

즉,
**노션처럼 편하지만 결과물은 이력서답게 안정적이어야 합니다.**

# 12. PDF 출력과의 관계

이력서 편집기가 자유로워질수록 PDF 출력은 더 어려워집니다.
그래서 편집 뷰와 출력 뷰를 분리해야 합니다.

## 권장 방식

* 편집용 리치 블록 문서
* 출력용 정규화 resume schema
* PDF 렌더러

즉, 편집기에서 바로 보이는 것을 그대로 캡처해서 PDF화하기보다,
**구조화 데이터로 정규화한 뒤 PDF 전용 템플릿으로 출력**하는 것이 좋습니다.

그 이유는 다음과 같습니다.

* 페이지 분할 제어 가능
* A4 안정성 확보
* 폰트/간격 일관성 유지
* ATS 친화성 유지

이 편집기 구조는 현재 기획한 거의 모든 기능과 연결됩니다.

실제로 질문 나온 문장을 편집기 안에서 바로 확인 가능

# 15. 권장 기능 정의

최종적으로는 아래처럼 정의하는 것이 좋습니다.

## 기능 요약

**이력서 편집 화면은 노션식 블록 편집 경험과 마크다운 입력 지원을 제공하되, 현재 프로젝트에 맞게 특정 문장/문단 선택, 인라인 포맷 변경, 코멘트, 질문 카드, 자동 질문 추천, 히트맵 오버레이, PDF 출력까지 연결되는 이력서 전용 워크스페이스로 설계한다.**

## 핵심 가치

* 개발자 친화적인 마크다운 입력
* 노션처럼 자연스러운 블록 편집
* 문서 위에서 직접 첨삭/질문/복기 가능
* 실전 면접 데이터와 이력서 연결 강화
* 제출용 출력 품질 보장

## 추천 우선순위

1. 블록 편집 + 마크다운 입력
2. 인라인 포맷 + 코멘트
3. 질문 카드 연결
4. AI 첨삭/자동 질문 추천
5. 히트맵 오버레이
6. 출력 최적화

정리하면,
이력서 편집 화면은 “노션처럼 쓸 수 있는 마크다운 편집기”가 되면 좋지만, 더 정확하게는
**“노션식 인터랙션을 가진 이력서 전용 면접 준비 편집기”**가 되어야 합니다.

원하시면 다음 단계로 이 부분만 따로 떼서
**이력서 편집 화면 상세 PRD** 형태로 더 구체화해드리겠습니다.
